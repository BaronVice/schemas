# 1. Список потребностей к стеку: функции под выбранные процессы

Для нашего кейса минимально достаточный стек должен поддерживать 3 выбранных процесса: 
1) проверка спроса и квалификация лидов; 
2) онбординг клиента и согласование первой поставки; 
3) приемка, упаковка, видеофиксация и отгрузка.

## 1.1. Потребности по процессу P1: проверка спроса и квалификация лидов

Что обязательно нужно стеку:

- принимать заявки из формы, звонка, мессенджера и ручного ввода;
- сохранять источник заявки: канал, UTM-метки, промокод, ручная пометка;
- присваивать уникальный ID лида;
- проверять дубли по телефону/контакту;
- хранить обязательные поля квалификации: регион, проблема, объем, хрупкость, готовность к первой поставке, допустимый тариф;
- вести лид по фиксированной воронке статусов;
- ставить задачу менеджеру, если лид неполный;
- считать конверсию по каналам и стоимость квалифицированного лида;
- поддерживать дашборд для решения «запускать / доработать / не запускать».

Нужные классы сервисов:

- форма захвата заявок;
- CRM;
- BI/аналитика;
- коммуникационный канал с отдельным номером.

## 1.2. Потребности по процессу P2: онбординг клиента и согласование первой поставки

Что обязательно нужно стеку:

- заводить карточку клиента из квалифицированного лида;
- хранить карточку первой поставки;
- фиксировать тариф, маркетплейс, объем первой партии, формат доставки, дату и время приемки;
- хранить статус «первая поставка согласована»;
- формировать заявку на приемку;
- отправлять клиенту подтверждение и оферту/условия;
- уведомлять склад о запланированном временном окне;
- фиксировать переносы, причины срыва и инициатора изменения.

Нужные классы сервисов:

- CRM с кастомными полями и статусами;
- шаблоны документов и облачное хранилище;
- календарь/напоминания;
- коммуникации с клиентом.

## 1.3. Потребности по процессу P3: приемка, упаковка, видеофиксация и отгрузка

Что обязательно нужно стеку:

- присваивать партии уникальный ID;
- связывать акт приема-передачи, фото/видео, чек-лист упаковки, отчет клиенту, счет и транспортную накладную в одной карточке партии;
- не давать перевести партию дальше без ссылки на видеофиксацию;
- хранить чек-лист упаковки и доказательства состояния товара;
- фиксировать плановую и фактическую дату отгрузки;
- фиксировать SLA и причину отклонения;
- хранить статус оплаты или условие постоплаты;
- регистрировать инциденты/претензии хотя бы в отдельной карточке или журнале.

Нужные классы сервисов:

- CRM или операционный контур для карточки партии;
- облачное медиахранилище;
- шаблоны актов, отчетов, накладных;
- бухгалтерский/платежный контур;
- аналитика по срокам, браку и полноте доказательного контура.

## 1.4. Сквозные потребности по данным, документам и безопасности

Стек также обязан:

- поддерживать ролевые доступы;
- включать 2FA для владельца, бухгалтера и IT-подрядчика;
- хранить документы по версиям;
- ограничивать срок действия ссылок;
- обеспечивать резервные копии и тест восстановления;
- логировать изменения статусов и ключевых полей;
- отделять персональные данные от операционных данных там, где это возможно.

## 1.5. Рекомендуемый минимально достаточный стек

Рекомендованный вариант для старта — «облачный, без кастомной разработки»:

1. Форма + простой лендинг
- для сбора заявок, UTM-меток, промокодов и первичных данных.

2. CRM
- единый центр для лидов, клиентов, первой поставки и партий;
- воронка статусов;
- обязательные поля;
- дедупликация;
- задачи менеджеру;
- карточка первой поставки;
- карточка партии.

3. Облачное хранилище документов и медиа
- оферта, акты, накладные, фото/видео, отчеты, архив версий.

4. BI-аналитика
- дашборды по лидам, каналам, согласованным первым поставкам, срокам отгрузки и полноте доказательного контура.

5. Отдельный рабочий номер + мессенджеры
- для входящих обращений и фиксации коммуникации.

6. Резервное копирование
- облачный бэкап + внешний зашифрованный SSD.

7. Легкий финансовый контур
- выставление счета, фиксация оплаты/постоплаты, выгрузка для бухгалтера.

## 1.6. Выбор сервисов: практичный вариант и альтернатива

Вариант А, рекомендованный для этого кейса

- формы и лендинг: сервисы экосистемы Яндекса или аналог;
- CRM: облачная CRM с гибкими полями и правами;
- хранилище: корпоративное облако с разграничением прав;
- аналитика: BI с простым подключением таблиц/CRM;
- резервирование: облачный бэкап + внешний SSD.

Почему это основной вариант:

- уже логически совпадает с архитектурой из материалов;
- быстрее внедряется за 1 неделю;
- не требует разработки;
- нормально покрывает must-требования.

Вариант Б, если нужен более «продажный» контур

- CRM делаем более ориентированной на продажи и коммуникации, а операционный блок по партии частично ведем в таблицах и облаке.

Когда выбирать:

- если в ближайшие месяцы главный фокус не складская дисциплина, а быстрый прозвон и конверсия лидов.

Но для нашего проекта я бы все же выбрал вариант А, потому что здесь критична не только продажа, но и доказуемый операционный контур партии.

# 2. Критерии выбора сервисов + анти-шаблоны

## 2.1. Критерии выбора сервисов

1. Покрытие must-требований
Сервис выбираем только если он поддерживает обязательные поля, статусы, роли, историю изменений, выгрузку данных и привязку файлов/ссылок.

2. Сквозная связность данных
Лид, клиент, первая поставка и партия должны связываться идентификаторами, а не жить в отдельных несвязанных файлах.

3. Быстрое внедрение
Стек должен запускаться за 5–7 рабочих дней без разработки собственной системы.

4. Работа в малой команде
Стек должен быть удобен для 2–5 пользователей, где владелец совмещает часть функций менеджера.

5. Низкая зависимость от ручного переноса данных
Желательны интеграции «форма → CRM → аналитика», иначе ошибки и задержки быстро съедят эффект.

6. Ролевой доступ и защита ПД
Обязательны разграничение прав, 2FA для критичных ролей, ограниченные ссылки и возможность не показывать лишние данные кладовщику/водителю.

7. Экспорт и резервирование
Данные должны выгружаться, а не быть заложниками одной платформы.

8. Масштабирование без боли
На старте достаточно простого облачного решения, но важно, чтобы позже можно было добавить WMS, API маркетплейсов, ЭДО или личный кабинет без разрушения модели данных.

9. Адекватная стоимость владения
Для теста спроса и первых 30 клиентов в месяц нельзя брать дорогой enterprise-контур.

## 2.2. Анти-шаблоны

1. «CRM нет, ведем все в мессенджерах и Excel»
Это ломает воронку, дедупликацию, контроль статусов и аналитику по каналам.

2. «Отдельно форма, отдельно таблица, отдельно папка, связей нет»
В этом случае нельзя собрать доказательный контур партии и тяжело разбирать претензии.

3. «Видео лежит только на телефоне сотрудника»
Такой подход убивает обещание прозрачности и создает риск потери доказательств.

4. «Один общий логин на всех»
Нельзя расследовать ошибки, нет персональной ответственности, высокий риск по ПД.

5. «Учет по клиенту есть, учета по партии нет»
Для фулфилмента это критическая ошибка: претензии, фото, акт и отгрузка должны жить на уровне партии.

6. «Сначала купим большую WMS/ERP»
Для пилота это преждевременно: дорого, долго и мешает быстро проверить спрос.

7. «Аналитику потом посчитаем вручную»
Тогда решение о запуске будет приниматься по ощущениям, а не по подтвержденной готовности клиентов к первой поставке.

8. «Ссылки на документы бессрочные и без пароля»
Это прямой риск утечки персональных данных и коммерческой информации.

9. «Утвержденные документы редактируем поверх старой версии»
Теряется юридическая и управленческая прослеживаемость.

# 3. Стоимость TCO на 6 месяцев: разовые и регулярные затраты + допущения

## 3.1. Допущения

- считаем только цифровой стек, а не весь бизнес целиком;
- не включаем аренду склада, грузовик, зарплаты водителей, упаковку, рекламу, ремонт;
- команда в цифровом контуре: 2–4 активных пользователя;
- стек облачный, без кастомной разработки;
- внедрение делает внешний IT-специалист/подрядчик в ограниченном объеме;
- цены указаны как рабочая оценка, потому что конкретный тариф зависит от выбранного бренда и числа пользователей.

## 3.2. Вариант 1 — пилотный минимум

Подходит, если задача — быстро подтвердить спрос и провести первые поставки с ручным, но управляемым контуром.

Разовые затраты:

- настройка CRM, воронки, полей, прав, дедупликации: 30–40 тыс. руб.;
- настройка облачного архива, структуры папок, прав доступа, шаблонов: 15–20 тыс. руб.;
- настройка дашборда и базовой аналитики: 10–15 тыс. руб.;
- внешний зашифрованный SSD для резервных копий: 8–12 тыс. руб.

Итого разово: 63–87 тыс. руб.

Регулярные затраты в месяц:

- CRM: 2–3 тыс. руб.;
- облачное хранилище и рабочие аккаунты: 1–2 тыс. руб.;
- рабочий номер/телефония: 1–1,5 тыс. руб.;
- дополнительное медиахранилище/бэкап: 0,5–1 тыс. руб.;
- техническая поддержка и мелкие доработки: 3–5 тыс. руб.

Итого в месяц: 7,5–12,5 тыс. руб.

Итого за 6 месяцев: 108–162 тыс. руб.

## 3.3. Вариант 2 — рабочий рекомендованный контур

Подходит, если сразу строим не только тест спроса, но и дисциплинированную первую операционную версию бизнеса.

Разовые затраты:

- CRM с карточками лида, клиента, первой поставки и партии: 40–50 тыс. руб.;
- интеграция формы, статусов, задач, дедупликации и уведомлений: 15–20 тыс. руб.;
- настройка облачного архива, прав, версионности, бэкапов: 20–25 тыс. руб.;
- настройка BI-дашбордов руководителя и операционного дашборда: 15–20 тыс. руб.;
- внешний SSD и базовое обучение сотрудников: 10–15 тыс. руб.

Итого разово: 100–130 тыс. руб.

Регулярные затраты в месяц:

- CRM: 3–5 тыс. руб.;
- облако и корпоративные аккаунты: 1,5–3 тыс. руб.;
- рабочий номер/телефония: 1–2 тыс. руб.;
- медиахранилище/бэкап: 1–1,5 тыс. руб.;
- техническая поддержка: 4–6 тыс. руб.

Итого в месяц: 10,5–17,5 тыс. руб.

Итого за 6 месяцев: 163–235 тыс. руб.

## 3.4. Какой вариант выбрать

Рекомендую вариант 2 как основной. Причина простая: вариант 1 дешевле, но на нем выше риск провала в трех местах:

- слабый контроль первой поставки;
- ручные ошибки в доказательном контуре партии;
- слабее защита ПД и управляемость доступа.

Если денег совсем мало, можно стартовать с вариантом 1, но сразу зафиксировать план перехода на вариант 2 к моменту появления стабильных первых клиентов.

# 4. Связка со смыслом: какие метрики успеха улучшает стек и за счет чего

Смысл стека в этом проекте не в «цифровизации ради цифровизации», а в том, чтобы улучшать именно те метрики, которые подтверждают спрос, управляемость первой поставки и качество основной услуги.

| Метрика успеха | Что в стеке на нее влияет | За счет чего улучшается |
|---|---|---|
| Доля квалифицированных лидов | форма + CRM + обязательные поля | лиды сразу собираются в едином формате, менеджер не теряет критичные поля, а неполные заявки автоматически дозаполняются |
| Конверсия «квалифицированный лид → согласованная первая поставка» | CRM-воронка + карточка первой поставки + напоминания | исчезает разрыв между «интерес есть» и «дата/объем/тариф согласованы» |
| Среднее время первого контакта | автоматическое создание лида + уведомление менеджеру | заявка не теряется в чате или таблице, менеджер быстрее выходит на клиента |
| Стоимость квалифицированного лида | фиксация UTM, каналов и расходов + BI | можно отключать слабые каналы и усиливать те, что дают не просто лиды, а готовых к пилоту клиентов |
| Доля первых поставок с полным комплектом данных до приезда товара | карточка первой поставки + обязательные поля + подтвержденный слот приемки | склад заранее знает дату, объем, формат доставки и требования к партии |
| Доля партий с полным доказательным контуром | ID партии + акт + фото/видео + чек-лист + отчет | прозрачность услуги становится не обещанием, а контролируемым стандартом |
| Среднее время от приемки до готовности к отгрузке | карточка партии + статусы + чек-лист упаковки | меньше хаоса и ручных уточнений, проще видеть узкие места в операции |
| Доля претензий по упаковке | чек-лист упаковки + медиафиксация + карточка инцидента | снижается вариативность качества и быстрее разбираются спорные случаи |
| Доля отгрузок в СЦ в согласованный срок | слот приемки + план/факт отгрузки + транспортные документы | появляется контроль SLA, видны причины задержек и ответственные |
| Доля завершенных услуг с подтвержденной выручкой | счет/условие постоплаты в карточке партии | цикл нельзя закрыть без финансового основания, поэтому выручка считается корректно |

## 4.1. Главный эффект по бизнес-смыслу

Стек прямо поддерживает ключевое обещание проекта:

- селлеру проще и быстрее передать товар;
- первая поставка меньше срывается;
- каждая партия прозрачна и доказуема;
- управленец принимает решение о запуске не по числу заявок, а по готовности клиентов реально передать товар;
- метрики из предыдущих разделов можно считать регулярно, а не вручную и постфактум.

## 4.2. Короткий вывод

Минимально достаточный цифровой стек для этого проекта — это не тяжелая WMS и не дорогая ERP, а связка из формы, CRM, облачного архива/медиахранилища, BI-аналитики, рабочего канала коммуникации и резервирования. Такой стек закрывает выбранные процессы из материалов, поддерживает требования по документам и ПД и, главное, позволяет считать реальные метрики: от стоимости квалифицированного лида до доли партий с полным доказательным контуром и своевременной отгрузкой.
